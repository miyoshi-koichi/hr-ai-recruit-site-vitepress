リポジトリの内容を詳しく確認できました。結論からお伝えします。
nepp-chan の概要
北海道音威子府村の「AI副村長」チャットアプリで、kayac社員がフルスクラッチで構築したものです。
技術スタック
レイヤー	技術
AIオーケストレーション	Mastra（エージェント、ツール、メモリ、RAG、MCP、ワークフロー、評価）
AI SDK	Vercel AI SDK + Google Gemini
バックエンド	Hono on Cloudflare Workers
フロントエンド	React + Vite + TailwindCSS on Cloudflare Pages
DB	Cloudflare D1 + Drizzle ORM
認証	WebAuthn（パスキー）
アーキテクチャ（server/src/mastra/配下）
agents/ — AIエージェント定義
tools/ — エージェントが使えるツール
memory/ — 会話記憶
mcp/ — MCP（Model Context Protocol）連携
workflows/ — ワークフロー定義
scorers/ — 応答品質の評価
data/ — RAG用データ処理
Difyの代替として使えるか？
結論: はい、十分可能です。 ただし性質が大きく異なります。
Dify vs nepp-chan方式の比較
観点	Dify	nepp-chan方式（Mastra + コード）
構築方法	ノーコード/ローコードのGUI	フルコード（TypeScript）
カスタマイズ性	GUIの範囲内	無制限
RAG	GUI設定	コードで自由に制御
エージェント	テンプレート的	完全カスタム可能
ワークフロー	ビジュアルエディタ	コードで柔軟に記述
モデル選択	多数対応	AI SDKで多数対応
デプロイ	Difyサーバー必要	Cloudflare Workersでエッジ配信
コスト	Difyライセンス＋サーバー代	Cloudflare（無料枠あり）+ AI API代のみ
学習コスト	低い	高い（TypeScript + Mastra理解が必要）
メンテナンス	Dify側が管理	自分で管理
自作で増やせる機能の例
nepp-chan方式にすると、Difyではやりにくい（または不可能な）ことが実現できます:
1. マルチチャネル対応
LINE、Slack、電話（音声AI）、対面端末など、好きなチャネルにAPIを接続可能
Difyだと基本的にWebチャットかAPI経由のみ
2. 高度なRAGパイプライン
チャンク戦略、リランキング、ハイブリッド検索をコードで細かく制御
ドメイン固有の前処理・後処理を自由に挟める
3. MCP連携
社内ツール、DB、外部APIをMCPサーバーとして接続
エージェントが社内システムを直接操作する仕組みが作れる
4. 複雑なワークフロー
条件分岐、並列処理、人間承認ステップなど、コードなら何でも可能
Difyのビジュアルワークフローでは表現しきれない複雑なロジックに対応
5. 評価・改善サイクル
nepp-chanにはscorers/（評価機能）が組み込まれている
応答品質を自動評価してモデルやプロンプトを改善するサイクルを回せる
6. 認証・ユーザー管理
WebAuthn（パスキー）、招待制など、自前で好きな認証方式を実装
Difyのユーザー管理の制約に縛られない
7. コスト最適化
Cloudflare Workers の無料枠 + 従量課金でスケール
Difyサーバーの維持費が不要
現実的な判断ポイント
nepp-chan方式に移行すべきケース:
Difyの制約にぶつかっている（カスタマイズ不足、ワークフローの限界など）
社内に TypeScript を書ける人がいる
長期的に機能を拡張していきたい
コストを抑えたい
Difyを続けるべきケース:
現状のDifyで十分要件を満たしている
エンジニアリソースが限られている
プロトタイプをすばやく回したい段階
nepp-chanのコードはAGPLv3ライセンスで公開されているので、社内（kayac）であればそのまま参考にしたりフォークして使えます。特に Mastra のセットアップパターン（エージェント定義、ツール統合、メモリ管理）は、自作AIチャットの良いリファレンスになります。
何か具体的に「こういう機能を追加したい」というのがあれば、nepp-chanのコードベースでどう実現できるか掘り下げられますよ。

両方の構造を詳しく調査しました。明確に比較できます。
結論: mirai-gikai をベースにすべき
理由を図にすると
あなたが欲しいもの:
  ✅ 会社説明のコンテンツページ
  ✅ テキスト選択 → AI質問
  ✅ チャットウィンドウ
  ✅ きれいなUI

nepp-chan が持っているもの:
  ✅ チャットウィンドウ（高機能）
  ✅ 管理ダッシュボード
  ❌ コンテンツページ → ゼロから作る必要あり
  ❌ テキスト選択機能 → ゼロから作る必要あり

mirai-gikai が持っているもの:
  ✅ コンテンツページ（テキスト表示）
  ✅ テキスト選択 → AI質問（完成済み）
  ✅ チャットウィンドウ（ストリーミング対応）
  ✅ コスト管理（日次上限）
  ✅ shadcn/ui の綺麗なUIコンポーネント
nepp-chanは「チャット専用アプリ」 — コンテンツページが一切ない。会社説明ページ + テキスト選択UIを全部自作する必要がある。
mirai-gikaiは「コンテンツ閲覧 + AIチャット」 — やりたいことの骨格がそのまま入っている。
mirai-gikai フォーク時の作業マップ
残すもの（そのまま使える）
機能	該当コード	理由
テキスト選択 → AI質問	features/bills/client/hooks/use-text-selection.ts + tooltip系	コア機能
チャットウィンドウ	features/chat/ 全体	そのまま使える
AI バックエンド	app/api/chat/route.ts + handle-chat-request.ts	モデル差し替えるだけ

「グローバルメニューがない」

「不要な部分を取り外したい」

整理してお伝えします。
インタビュー機能とは
「AIがユーザーに質問して意見を引き出す」逆方向のチャット機能です。
通常のチャット: ユーザーが質問 → AIが回答 インタビュー: AIが質問 → ユーザーが回答 → AIがさらに深掘り → レポート自動生成
具体的なフロー
1. 法案ページで「インタビューを始める」
2. AIが最初の質問を自動生成
   「この法案について、どう思いますか？」
3. ユーザーが回答（テキスト or クイックリプライボタン）
4. AIが深掘り（3〜5問）
   「具体的にどの点が気になりましたか？」
5. 完了 → AIがレポートカードを自動生成
   - 要約 / 賛否 / 立場 / 意見3点 / スコア
6. ユーザーが確認して「提出」
裏側の技術が面白い
LLM 2つ同時使用: 「ファシリテーターLLM」が会話の進行度を判定し、「チャットLLM」が実際の質問を生成
ステージ管理: chat → summary → complete の3段階を自動遷移
採用サイトへの転用可能性
これ、「AIカジュアル面談」にそのまま使えます:
mirai-gikai	採用サイト版
法案	募集ポジション
「この法案についてどう思う？」	「カヤックに興味を持ったきっかけは？」
深掘り質問	「具体的にどんなプロジェクトに惹かれました？」
レポート（賛否・立場）	プロフィールカード（志望度・経験・強み3点）
提出	エントリー送信
ただし初期リリースには不要。まずはコンテンツ + テキスト選択チャットを固めてから、第2フェーズで検討する機能です。
nepp-chan から借用すると良い機能
TOP3（おすすめ順）
1位: ツールUI群（Chart / Table / Timeline / ChoiceButtons） — 移植: 簡単
テキストだけのチャットと、これがあるチャットでは体験が全く違います。
例:
・「選考フローは？」→ タイムラインで表示
・「どんな職種がある？」→ テーブルで一覧表示
・「社員の技術スタック分布は？」→ 円グラフで表示
・「どの事業に興味ある？」→ 選択ボタンで聞く
nepp-chanでは @assistant-ui/react のツールUI登録の仕組みで実装されていて、UIコンポーネントをコピーするだけで動きます。
2位: フィードバック機能 — 移植: 中
メッセージごとに Good/Bad/Idea の3段階評価 + カテゴリ + コメントを送れる仕組み。「AIが変な回答してないか」を運用で把握するために重要。DBテーブルの新設が必要ですが、UIはコピーで移植可能。
3位: Working Memory（ユーザー記憶） — 移植: 簡単
再訪問した人に「前回エンジニア職に興味があるとおっしゃっていましたね」と言える機能。Mastraのメモリ設定にスキーマを追加するだけ。ただし mirai-gikai ベースで作るなら Mastra は使わないので、別の方法で実装する必要があります。
転用しなくていい機能
緊急報告（クマ出没通報とかの機能）
天気エージェント
Google検索連携（採用情報は自社ナレッジで完結すべき）
ペルソナ分析ダッシュボード（ユーザー増えてからでいい）
まとめ: 構成の全体像
Phase 1（まずこれ）
  ベース: mirai-gikai フォーク
  ├── コンテンツページ（会社説明）
  ├── テキスト選択 → AI質問（mirai-gikaiそのまま）
  ├── チャットウィンドウ（mirai-gikaiそのまま）
  ├── グローバルナビ追加
  └── デザイン変更

Phase 2（余裕ができたら）
  nepp-chanから借用:
  ├── ツールUI（Chart/Table/Timeline/ChoiceButtons）
  ├── フィードバック機能
  └── Working Memory

Phase 3（将来）
  mirai-gikaiのインタビュー機能を転用:
  └── AIカジュアル面談（エントリー機能）

おっしゃる通りです。手段から入るのではなく、まずアイデアを出し切りましょう。
両リポジトリの技術的な能力を踏まえて、「採用を改善/改革する」アイデアを発散させます。
両リポジトリが持つ「武器」の棚卸し
まずこれを頭に入れた上で発想します:
mirai-gikaiの武器:
  ・テキスト選択 → AI質問
  ・AIインタビュー（AIが質問する側になる）
  ・構造化レポート自動生成
  ・難易度切り替え（説明の深さ調整）
  ・コンテンツページ + チャット統合UI
  ・コスト管理・日次上限
  ・匿名認証（登録不要で使える）
  ・プロンプト管理（Langfuse）
  ・テレメトリ（利用分析）

nepp-chanの武器:
  ・ツールUI（グラフ/テーブル/タイムライン/選択ボタン）
  ・Working Memory（ユーザーを覚える）
  ・マルチエージェント委譲
  ・フィードバック収集 + 管理ダッシュボード
  ・ペルソナ自動蓄積・分析
  ・RAG + リランキング（高精度な情報検索）
  ・ナレッジ管理UI
  ・ドキュメント変換（PDF/画像 → テキスト）
  ・Eval（AI回答品質の自動テスト）
  ・キャラクター性のあるAI（「ねっぷちゃん」）
アイデア一覧
A. 候補者が「会社を知る」フェーズ
A1. テキスト選択で即質問できる採用サイト 会社説明を読んでいて「ブレスト文化って具体的に何するの？」と思ったら、その場で選択して聞ける。mirai-gikaiのコア機能そのまま。
A2. 「聞きにくいこと」を聞けるAI 「残業多い？」「給料いくら？」「離職率は？」— 人事には聞きにくいことをAIなら気軽に聞ける。匿名認証なので誰が聞いたかわからない。RAGに会社の実態データ（平均残業時間、有給消化率など）を入れておけば正直に答えられる。
A3. 説明の深さ切り替え mirai-gikaiの「難易度切り替え」を転用。学生向け/経験者向けで説明の粒度を変える。「エンジニア向けに詳しく」を押すと技術スタックの詳細が出てくる、「ざっくり知りたい」を押すと概要だけ、など。
A4. AI社員ロールプレイ nepp-chanの「キャラクター性」+ RAGを応用。実際の社員インタビュー記事をナレッジに入れて「エンジニアの○○さんに聞いてみる」のように、社員の視点で答えるAI。本人が答えているわけではないが、社員の声ベースで回答する。
A5. 事業紹介のインタラクティブ探索 ChoiceButtons（選択ボタン）で「どの事業に興味ある？」→ 選んだ事業の詳細をChart/Tableで可視化。「面白プロデュース事業の実績」を円グラフで見せたり、「過去のプロジェクト」をタイムラインで表示したり。テキストの壁ではなく対話的に会社を知れる。
B. 候補者が「応募を検討する」フェーズ
B1. ポジションマッチングAI AIが「どんな仕事がしたい？」「得意な技術は？」「働き方の希望は？」を ChoiceButtons で聞いていき、最適なポジションを提案。nepp-chanのマルチエージェント委譲で、職種ごとの専門エージェントに振り分け。
B2. 「自分に合うか」診断 mirai-gikaiのインタビュー機能を転用。AIが5問くらい質問して、カルチャーフィット度をレポート化。「あなたはカヤックの○○な部分と相性が良さそうです」とフィードバック。エントリーへの後押しになる。
B3. 選考フローナビゲーター 「エンジニアの選考フローは？」と聞くと TimelineUI で「書類選考 → 技術面接 → カルチャー面接 → 最終」をビジュアル表示。各ステップの詳細も聞ける。
B4. 給与・待遇シミュレーター 「エンジニア5年目だとどのくらい？」に対して、公開可能な範囲でTable/Chartで提示。福利厚生の一覧もテーブルで見やすく。
C. 候補者が「応募する」フェーズ
C1. AIカジュアル面談 → そのままエントリー mirai-gikaiのインタビュー機能の最大の転用先。AIと3〜5分話すだけで、志望動機・経験・強みが構造化レポートになり、それがそのままエントリーシート代わりに提出される。「エントリーシート書くのだるい」を解消。
C2. ポートフォリオ対話 nepp-chanのドキュメント変換（PDF → テキスト）を応用。候補者がポートフォリオURLやPDFを送ると、AIがそれを読んで「このプロジェクトについてもう少し教えてください」と深掘り。結果がレポートになる。
C3. Working Memoryで再訪問フォロー 一度サイトを訪れた人が戻ってきたとき、「前回ゲーム事業に興味を持っていましたね。その後新しいプロジェクトが始まりましたよ」と文脈を覚えている。検討中の候補者の背中を押せる。
D. 社内（採用チーム側）の改善
D1. 「何を聞かれているか」分析ダッシュボード nepp-chanのペルソナ分析を転用。候補者がAIに何を質問しているかを集計・可視化。「給与の質問が40%」「リモートワーク可否が30%」→ 採用ページのどこを充実させるべきかがデータでわかる。
D2. AI回答品質の監視 フィードバック機能で候補者が「この回答は役に立った/立たなかった」を評価。Bad評価が多い回答パターンを特定して、ナレッジやプロンプトを改善するサイクルを回せる。
D3. ナレッジ管理の自動化 会社の制度変更、新プロジェクト、新オフィスなどの情報をナレッジ管理UIから更新。ドキュメント変換で社内資料（PDF）を即座にAIの知識に追加。
D4. 採用FAQの自動生成 チャットログから「よく聞かれる質問TOP20」を自動抽出。それを採用サイトのFAQページに自動反映。コンテンツが勝手に充実していく。
D5. Evalで回答品質を担保 nepp-chanのEval機能を転用。「この質問にはこう答えるべき」というテストケースを用意して、プロンプト変更時にAIの回答品質が落ちていないか自動チェック。
E. 既存の枠を超えるもの
E1. 「面白法人ブレストAI」 カヤックの最大の文化である「ブレスト」をAI相手に体験できる。お題を出すとAIが一緒にアイデア出しをしてくれる。候補者が「この会社の考え方、合うかも」と感じる体験型コンテンツ。採用活動であると同時にブランディング。
E2. AIリファラル支援 社員が知人に会社を紹介するとき、「うちの会社のここが面白いよ」をAIがその人に合わせて説明してくれるリンクを生成。紹介された人がリンクを開くと、AIが1対1で会社説明してくれる。
E3. 内定者フォローAI 内定から入社までの期間、内定者の不安や質問にAIが24時間対応。Working Memoryで「入社前に引っ越しが必要と言ってたけど進捗どう？」みたいな継続的なフォロー。人事の負荷を大幅に削減。
E4. 選考体験アンケートAI 選考を受けた候補者（合否問わず）に対して、インタビュー機能で「選考体験はどうでしたか？」をAIがヒアリング。構造化レポートで改善点が自動まとまる。
E5. 採用イベント × AI 会社説明会や採用イベントで参加者がスマホからAIに質問できる。その場で聞きにくいことも匿名で聞ける。イベント後のフォローもWorking Memoryで継続。

GitHub で Archive 化するなら、名前に _archive は不要です。
理由:
Archive したリポジトリには GitHub が自動で "This repository has been archived" のバッジを表示する
リポジトリ一覧でもグレーアウトされて「これはもう使ってない」と一目でわかる
_archive を付けると二重表現になる
おすすめの命名
旧: kayac-recruit-site → そのまま。GitHub Archive 化するだけ
新: hr-ai-recruit-site  ← mirai-gikai フォーク
旧リポジトリを hr-ai-recruit-site 系の名前に揃える必要もないと思います。別物なので。旧はそのまま触らず Archive、新しい名前で新しく始めるのが一番すっきりします。
ただ、もし旧リポジトリの名前も揃えたいなら kayac-recruit-site-vitepress とか中身がわかる名前の方が _archive より良いです。
この方針でよければ:
現リポジトリを GitHub で Archive 化
